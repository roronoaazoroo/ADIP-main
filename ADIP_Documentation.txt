1. EXECUTIVE SUMMARY
================================================================================

1.1 Purpose
-----------
The Azure Drift Intelligence Platform (ADIP) is a cloud-native monitoring and
remediation system built entirely on Microsoft Azure. It continuously monitors
Azure infrastructure resources, detects configuration drift from a defined
golden baseline, and provides real-time visibility and auto-remediation
capabilities through an interactive web dashboard.

1.2 Problem Statement
---------------------
In enterprise cloud environments, infrastructure configurations change
frequently due to manual interventions, automated deployments, or policy
violations. These untracked changes — known as "configuration drift" — can
lead to security vulnerabilities, compliance failures, performance degradation,
and unpredictable system behaviour.

Traditional monitoring tools detect failures after they occur. ADIP takes a
proactive approach: it detects the moment a resource deviates from its
approved baseline and immediately surfaces that information to operators.

1.3 Goals
---------
- Detect configuration drift in Azure resources in real time
- Compare live resource state against a stored golden baseline
- Provide a visual diff of what changed, when, and by whom
- Enable one-click remediation to revert resources to their baseline state
- Stream live resource change events to the dashboard without page refresh
- Store all drift history in Cosmos DB for audit and compliance

1.4 Key Features
----------------
- Real-time change feed via Event Grid → Storage Queue → Socket.IO
- JSON diff engine comparing live ARM state vs Cosmos DB baseline
- Severity classification (Critical / High / Medium / Low)
- Golden baseline management — immutable reference configuration
- One-click remediation — reverts live Azure resource to baseline via ARM API
- Modular Node.js Express backend with full REST API
- Azure Function for automated drift detection triggered by Event Grid
- Logic App workflow orchestrating the event-to-function pipeline

2. ARCHITECTURE OVERVIEW
================================================================================

2.1 High-Level Architecture
----------------------------
The system is composed of five layers:

  [User Browser]
       |
       v
  [React Frontend] <-----> [Socket.IO / Real-time feed]
       |                          ^
       v                          |
  [Node.js Express API]           |
       |                    [Queue Poller]
       |                          ^
       v                          |
  [Azure Cosmos DB]    [Azure Storage Queue]
       |                          ^
       v                          |
  [Azure Function]       [Azure Event Grid]
       ^                          ^
       |                          |
  [Logic App] <---------- [ARM Resource Changes]

2.2 Data Flow (Top-Down)
-------------------------
Step 1: A user or automated process modifies an Azure resource (e.g., changes
        a storage account's TLS setting or network ACL).

Step 2: Azure Resource Manager emits a ResourceWriteSuccess or
        ResourceDeleteSuccess event to the Event Grid Topic (adip-eg-topic).

Step 3: Event Grid fans out the event to two subscribers:
        a) Azure Storage Queue (adip-rg-changes subscription) — for the
           real-time change feed
        b) Logic App HTTP trigger (adip-logic-sub) — for drift detection

Step 4: The Logic App parses the event, extracts resourceId and subscriptionId,
        and calls the Azure Function (detectDrift) via HTTP POST.

Step 5: The Azure Function:
        - Fetches the current live configuration from ARM using Azure SDK
        - Fetches the golden baseline from Cosmos DB (baselines container)
        - Performs a deep JSON diff between the two states
        - Classifies severity based on which fields changed
        - Stores the drift record in Cosmos DB (drift-records container)
        - Notifies the Express API to push the event via Socket.IO

Step 6: The Express API's Queue Poller reads messages from the Storage Queue
        every 5 seconds and emits 'resourceChange' events via Socket.IO to
        all connected frontend clients subscribed to the relevant scope.

Step 7: The React frontend receives the Socket.IO event and appends it to the
        Live Activity feed in real time — no page refresh required.

Step 8: The user can click "View diff" to navigate to the Comparison Page,
        which shows a side-by-side JSON diff of the golden baseline vs the
        current live state, with severity classification and field-level changes.

Step 9: If the user clicks "Remediate to Baseline", the Express API calls the
        ARM API to PUT the baseline configuration back onto the live resource,
        reverting the drift.

3. DETAILED COMPONENT BREAKDOWN
================================================================================

3.1 Frontend — React + Vite
----------------------------
WHAT IT IS:
  A single-page application (SPA) built with React 18 and bundled with Vite.
  It provides the user interface for resource selection, configuration viewing,
  drift comparison, and real-time monitoring.

WHY REACT:
  React's component model and virtual DOM make it ideal for dashboards with
  frequently updating data. Vite provides near-instant hot module replacement
  during development and optimised production builds.

KEY PAGES:
  - LoginPage: Entry point, routes directly to dashboard on button click
  - DashboardPage: Resource selection, configuration JSON viewer, live feed
  - ComparisonPage: Side-by-side baseline vs live state diff viewer

KEY HOOKS:
  - useAzureScope: Fetches subscriptions, resource groups, and resources from
    the Express API. Falls back to demo data if backend is unreachable.
  - useDriftSocket: Manages Socket.IO connection. Listens for 'resourceChange'
    events and appends them to the Live Activity feed.
  - DashboardContext: React context that persists dashboard state (selected
    resource, config data, live events) across page navigation.

HOW IT WORKS:
  1. User selects subscription → resource group → resource
  2. Clicks Submit → Express API fetches live ARM configuration
  3. JSON is rendered in an interactive collapsible tree (JsonTree component)
  4. Socket.IO connects and subscribes to the selected scope room
  5. Any resource changes in that scope appear in the Live Activity feed
  6. User clicks Compare → navigates to ComparisonPage with live state
  7. ComparisonPage fetches baseline from Cosmos DB and diffs against live state

3.2 Backend — Node.js Express API
-----------------------------------
WHAT IT IS:
  A RESTful API server built with Express.js (Node.js). It acts as the
  middleware between the React frontend and Azure services (Cosmos DB, ARM,
  Storage Queue).

WHY NODE.JS + EXPRESS:
  Node.js is non-blocking and event-driven, making it well-suited for I/O
  heavy workloads like polling queues and proxying ARM API calls. Express
  provides a minimal, flexible routing layer. Using Node.js for both the
  backend and Function App ensures a single language across the stack.

API ENDPOINTS:
  GET  /api/subscriptions                              — List Azure subscriptions
  GET  /api/subscriptions/:id/resource-groups          — List resource groups
  GET  /api/subscriptions/:id/resource-groups/:rg/resources — List resources
  GET  /api/configuration                              — Fetch live ARM config
  POST /api/scan/start                                 — Fetch config + return
  GET  /api/drift-events                               — Query drift history
  GET  /api/baselines                                  — Fetch golden baseline
  POST /api/baselines                                  — Save new baseline
  POST /api/compare                                    — Manual drift detection
  POST /api/monitor/start                              — Start server monitoring
  POST /api/monitor/stop                               — Stop server monitoring
  POST /api/remediate                                  — Revert resource to baseline
  POST /api/seed-baseline                              — Seed dummy test baseline
  POST /internal/drift-event                           — Internal: Function → Socket.IO

REAL-TIME (Socket.IO):
  The Express server creates a Socket.IO server on the same HTTP port.
  Clients emit 'subscribe' with their scope (subscriptionId + resourceGroup).
  The server joins them to a room named "subscriptionId:resourceGroup".
  When the queue poller dequeues a change event, it emits 'resourceChange'
  to the matching room — all subscribed clients receive it instantly.

QUEUE POLLER (queuePoller.js):
  Runs on server startup. Every 5 seconds, it calls receiveMessages() on the
  Azure Storage Queue, parses the base64-encoded Event Grid payload, extracts
  the resource change details, emits them via Socket.IO, and deletes the
  processed messages from the queue.

3.3 Azure Event Grid
---------------------
WHAT IT IS:
  A fully managed event routing service that reacts to state changes in Azure
  resources and routes events to configured endpoints.

WHY EVENT GRID:
  Event Grid is the native Azure eventing backbone. It integrates directly
  with ARM (Azure Resource Manager) as an event source, meaning any resource
  write or delete in a subscription or resource group automatically generates
  an event — no polling or custom instrumentation required.

HOW IT WORKS IN ADIP:
  - Source: Azure Resource Group (rg-adip) — catches all ARM operations
  - Event types: ResourceWriteSuccess, ResourceDeleteSuccess,
    ResourceActionSuccess
  - Two subscriptions:
    a) adip-rg-changes → Storage Queue (for real-time feed)
    b) adip-logic-sub  → Logic App HTTP trigger (for drift detection)

WHY NOT EVENT HUBS:
  Event Hubs is designed for high-throughput telemetry streaming (millions of
  events/sec). For infrastructure change events (low volume, high importance),
  Event Grid is more appropriate — it provides push-based delivery, filtering,
  dead-lettering, and native ARM integration without requiring a consumer group
  model.

3.4 Azure Storage Queue
------------------------
WHAT IT IS:
  A simple, durable message queue service in Azure Storage. Messages are
  stored until consumed and support visibility timeouts for reliable processing.

ROLE IN ADIP:
  Acts as the buffer between Event Grid and the Express API's queue poller.
  Event Grid writes change events here; the poller reads and deletes them.
  This decouples the event source from the consumer — if the Express API is
  temporarily down, events accumulate in the queue and are processed when
  it restarts.

WHY STORAGE QUEUE OVER SERVICE BUS:
  Storage Queue is simpler, cheaper, and sufficient for this use case.
  Service Bus adds features like topics, sessions, and transactions — none
  of which are needed here. Storage Queue integrates natively with Event Grid
  as a delivery endpoint.

3.5 Azure Logic App
--------------------
WHAT IT IS:
  A low-code workflow automation service. In ADIP, it acts as the orchestrator
  between Event Grid and the Azure Function.

WORKFLOW:
  Trigger: HTTP Request (receives Event Grid webhook POST)
  Step 1:  Check if event is a SubscriptionValidationEvent → respond with
           validationCode (required by Event Grid for webhook registration)
  Step 2:  Extract resourceUri and subscriptionId from event data
  Step 3:  HTTP POST to Azure Function (detectDrift) with extracted data
  Step 4:  Return Function response

WHY LOGIC APP:
  Logic Apps provide a visual, auditable workflow with built-in retry logic,
  run history, and monitoring. They decouple the event routing logic from the
  Function code, making it easy to add steps (e.g., send email alert, create
  ticket) without modifying the Function.

3.6 Azure Function — detectDrift
----------------------------------
WHAT IT IS:
  A serverless Node.js function (HTTP trigger) that performs the core drift
  detection logic.

HOW IT WORKS:
  Input:  { resourceId, subscriptionId }
  Step 1: Validate input — return 400 if missing
  Step 2: Handle Event Grid validation handshake (return validationCode)
  Step 3: Parse ARM resource ID to extract resourceGroup, provider, type, name
  Step 4: Use ResourceManagementClient (Azure SDK) to GET live resource config
  Step 5: Query Cosmos DB baselines container for active baseline (resourceId)
  Step 6: Strip volatile fields (etag, _ts, provisioningState, systemData)
  Step 7: Run deep-diff between stripped baseline and live state
  Step 8: Classify severity:
          - CRITICAL: any field was deleted
          - HIGH:     security-sensitive path changed (networkAcls, encryption)
          - MEDIUM:   more than 5 fields changed
          - LOW:      minor changes
  Step 9: If drift detected, save record to drift-records container
  Step 10: POST to Express API /internal/drift-event to trigger Socket.IO push
  Output: { drifted, differences, severity, liveState, baselineState }

WHY SERVERLESS:
  The drift detection function runs only when triggered — no idle compute cost.
  Azure Functions scale automatically and integrate natively with Event Grid,
  Logic Apps, and Cosmos DB.

3.7 Azure Cosmos DB
--------------------
WHAT IT IS:
  A globally distributed, multi-model NoSQL database. ADIP uses the SQL API
  (document model) with two containers.

CONTAINERS:
  baselines (partition key: /resourceId)
    - Stores golden baseline configurations
    - Each document has active: true/false to track the current baseline
    - Immutable once set — remediation reads from here, never overwrites

  drift-records (partition key: /subscriptionId)
    - Stores every detected drift event
    - Contains: resourceId, differences[], severity, liveState, baselineState,
      detectedAt, changeCount

WHY COSMOS DB OVER SQL:
  Drift records and baselines are JSON documents of variable structure
  (different resource types have different property schemas). A document
  database handles this naturally without schema migrations. Cosmos DB also
  provides single-digit millisecond reads, which is critical for the
  comparison page load time.

3.8 Azure Identity & Authentication
-------------------------------------
WHAT IT IS:
  The @azure/identity SDK provides DefaultAzureCredential, which automatically
  selects the best available credential source.

HOW IT WORKS IN ADIP:
  - Local development: uses the logged-in Azure CLI credential (az login)
  - Azure Function / App Service: uses the system-assigned Managed Identity
  - No secrets stored in code — credentials are resolved at runtime

MANAGED IDENTITY PERMISSIONS REQUIRED:
  - Reader role on the subscription (to fetch resource configurations)
  - Contributor role on rg-adip (to PUT remediated configurations)
  - Cosmos DB Data Contributor (to read/write Cosmos documents)

3.9 Monitoring & Logging
-------------------------
WHAT IT IS:
  Azure Application Insights is automatically provisioned with the Function App.
  It captures invocation logs, exceptions, dependencies, and performance metrics.

IN ADIP:
  - Function App: context.log() calls are captured in Application Insights
  - Express API: console.log() — can be extended with Winston + App Insights SDK
  - Live Activity Feed: provides operator-level visibility in the UI itself
  - Drift records in Cosmos DB serve as an audit log of all detected changes

4. STEP-BY-STEP FLOW EXPLANATION
================================================================================

SCENARIO: An operator changes the minimumTlsVersion of a storage account
from TLS1_2 to TLS1_0 in the Azure Portal.

Step 1  [ARM]         Azure Resource Manager processes the PUT operation and
                      marks it as ResourceWriteSuccess.

Step 2  [Event Grid]  ARM publishes the event to the Event Grid topic
                      (adip-eg-topic). Event contains:
                      - eventType: Microsoft.Resources.ResourceWriteSuccess
                      - data.resourceUri: /subscriptions/.../storageAccounts/adipstore001
                      - data.subscriptionId: 8f461bb6-...
                      - data.operationName: Microsoft.Storage/storageAccounts/write
                      - data.claims.name: operator@company.com

Step 3a [Queue]       Event Grid delivers the event to the Storage Queue
                      (resource-changes). Message is base64-encoded JSON.

Step 3b [Logic App]   Event Grid also delivers to the Logic App HTTP trigger.
                      Logic App extracts resourceUri + subscriptionId and
                      calls the Azure Function via HTTP POST.

Step 4  [Queue Poll]  Express API queue poller (running every 5s) dequeues
                      the message, parses it, and emits 'resourceChange' via
                      Socket.IO to room "8f461bb6-...:rg-adip".

Step 5  [Frontend]    React frontend (subscribed to that room) receives the
                      'resourceChange' event. Live Activity feed shows:
                      "MODIFIED — adipstore001  by operator@company.com"
                      — appears instantly, no refresh needed.

Step 6  [Function]    Azure Function fetches live config of adipstore001 from
                      ARM. Fetches baseline from Cosmos DB. Diffs the two:
                      Finds: properties.minimumTlsVersion changed TLS1_2 → TLS1_0
                      Severity: HIGH (security-sensitive path)

Step 7  [Cosmos DB]   Function saves drift record to drift-records container.

Step 8  [Socket.IO]   Function POSTs to /internal/drift-event on Express API.
                      Express emits 'driftEvent' to the room.

Step 9  [User Action] Operator clicks the change in the Live Activity feed →
                      navigates to Comparison Page.

Step 10 [Comparison]  Page fetches baseline from Cosmos DB. Shows side-by-side:
                      Golden Baseline: minimumTlsVersion = "TLS1_2"
                      Live State:      minimumTlsVersion = "TLS1_0"
                      Severity badge: HIGH

Step 11 [Remediate]   Operator clicks "Remediate to Baseline". Express API
                      calls ARM PUT with the baseline configuration.
                      Storage account is reverted to TLS1_2.
                      Success panel shows: "1 field reverted to golden baseline"

5.1 Azure Function — Drift Detection Core (Node.js)
----------------------------------------------------

// detectDrift/index.js
module.exports = async function (context, req) {

  // Handle Event Grid validation handshake
  const body = req.body
  if (Array.isArray(body) &&
      body[0]?.eventType === 'Microsoft.EventGrid.SubscriptionValidationEvent') {
    context.res = {
      status: 200,
      body: { validationResponse: body[0].data.validationCode }
    }
    return
  }

  // Extract resource details from Event Grid payload
  const eventData = Array.isArray(body) ? body[0]?.data : body
  const { resourceId, subscriptionId } = eventData || {}

  // Fetch live configuration from ARM
  const credential = new DefaultAzureCredential()
  const armClient  = new ResourceManagementClient(credential, subscriptionId)
  const liveRaw    = await armClient.resources.get(
    resourceGroupName, provider, '', type, name, '2021-04-01'
  )

  // Fetch golden baseline from Cosmos DB
  const { resources: baselines } = await baselineContainer.items.query({
    query: 'SELECT TOP 1 * FROM c WHERE c.resourceId = @rid AND c.active = true',
    parameters: [{ name: '@rid', value: resourceId }]
  }).fetchAll()

  // Compute diff (strip volatile fields first)
  const differences = diff(strip(baselines[0].resourceState), strip(liveRaw)) || []
  const severity    = classifySeverity(differences)

  // Save drift record if drift detected
  if (differences.length > 0) {
    await driftContainer.items.create({ id: `drift-${Date.now()}`, ...record })
    // Notify Express API to push via Socket.IO
    await fetch(`${process.env.EXPRESS_API_URL}/internal/drift-event`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(record)
    })
  }

  context.res = { status: 200, body: record }
}

5.2 Storage Queue Poller (Node.js)
-----------------------------------

// services/queuePoller.js
function startQueuePoller() {
  const client   = getQueueClient()
  const interval = parseInt(process.env.QUEUE_POLL_INTERVAL_MS || '5000')

  setInterval(async () => {
    // Dequeue up to 32 messages
    const response = await client.receiveMessages({
      numberOfMessages: 32,
      visibilityTimeout: 30
    })

    for (const msg of response.receivedMessageItems) {
      const event = parseMessage(msg)  // base64 decode + JSON parse
      if (event) {
        // Broadcast to subscribed frontend clients
        const room = `${event.subscriptionId}:${event.resourceGroup}`
        global.io.to(room).emit('resourceChange', event)

        // Delete processed message
        await client.deleteMessage(msg.messageId, msg.popReceipt)
      }
    }
  }, interval)
}

5.3 Express API — Remediate to Baseline
-----------------------------------------

// routes/remediate.js
router.post('/remediate', async (req, res) => {
  const { subscriptionId, resourceGroupId, resourceId } = req.body

  // Fetch golden baseline (immutable source of truth)
  const baseline = await getBaseline(subscriptionId, resourceId)
  const baselineState = strip(baseline.resourceState)

  // Apply baseline state back to Azure via ARM PUT
  await armClient.resources.beginCreateOrUpdateAndWait(
    resourceGroupName, provider, '', type, name, '2021-04-01',
    { ...baselineState, location: baseline.resourceState.location }
  )

  res.json({ remediated: true, resourceId, changeCount: differences.length })
})

// hooks/useDriftSocket.js
socket.on('resourceChange', (event) => {
  // Filter to selected scope only
  const matchesSub = event.subscriptionId === scope.subscriptionId
  const matchesRG  = !scope.resourceGroup ||
                     event.resourceGroup === scope.resourceGroup
  if (matchesSub && matchesRG) {
    setChangeEvents(prev => [{
      ...event,
      _clientId:   `${event.eventId}-${Math.random()}`,
      _receivedAt: new Date().toLocaleTimeString()
    }, ...prev].slice(0, 200))
  }
})

5.5 Cosmos DB — Baseline Query
--------------------------------

// services/cosmosService.js
async function getBaseline(subscriptionId, resourceId) {
  const { resources } = await baselineContainer().items.query({
    query: `SELECT TOP 1 * FROM c
            WHERE c.resourceId = @rid
            AND c.subscriptionId = @sub
            AND c.active = true
            ORDER BY c._ts DESC`,
    parameters: [
      { name: '@rid', value: resourceId },
      { name: '@sub', value: subscriptionId }
    ]
  }).fetchAll()
  return resources[0] || null
}

6. DESIGN DECISIONS & TRADE-OFFS
================================================================================

6.1 Event Grid vs Polling
--------------------------
CHOSEN: Event Grid push model
ALTERNATIVE: Periodic polling of ARM API
REASON: Polling introduces latency and unnecessary API calls. Event Grid
delivers events within seconds of the change occurring, with no wasted
requests. It also scales to thousands of resources without additional cost.

6.2 Storage Queue vs Service Bus vs Event Hubs
-----------------------------------------------
CHOSEN: Azure Storage Queue
ALTERNATIVE: Service Bus (more features), Event Hubs (high throughput)
REASON: Infrastructure change events are low-volume and high-importance.
Storage Queue provides durable, ordered delivery with visibility timeouts
at minimal cost (~$0.004 per 10,000 operations). Service Bus and Event Hubs
add unnecessary complexity and cost for this volume.

6.3 Cosmos DB vs Azure SQL
---------------------------
CHOSEN: Cosmos DB (SQL API / document model)
ALTERNATIVE: Azure SQL Database (relational)
REASON: Resource configurations are JSON documents with variable schemas
(a VM config looks nothing like a Key Vault config). Cosmos DB stores them
natively without schema migrations. It also provides single-digit millisecond
reads critical for the comparison page.

6.4 Node.js vs Python (FastAPI)
---------------------------------
CHOSEN: Node.js (Express + Azure Functions)
ALTERNATIVE: Python FastAPI
REASON: The frontend uses JavaScript/React. Using Node.js for the backend
and Function App creates a single-language stack, simplifying onboarding,
tooling, and dependency management. The Azure SDK for JavaScript is
feature-complete and well-maintained.

6.5 Socket.IO vs Azure SignalR Service
----------------------------------------
CHOSEN: Socket.IO (self-hosted on Express server)
ALTERNATIVE: Azure SignalR Service (managed)
REASON: For local development and small-scale deployments, Socket.IO running
on the Express server is simpler and free. Azure SignalR Service is the
recommended path for production at scale (handles connection management,
scaling, and Azure AD integration). The architecture supports switching to
SignalR with minimal code changes.

6.6 Immutable Golden Baseline
-------------------------------
DESIGN DECISION: The golden baseline is never overwritten by the "Remediate"
action. Remediation reads the baseline and applies it to the live resource.
REASON: The baseline is the source of truth. If remediation could overwrite
it, a drifted state could accidentally become the new baseline, defeating
the purpose of drift detection.

7. ERROR HANDLING & RELIABILITY
================================================================================

7.1 Event Grid Retry Policy
-----------------------------
Event Grid automatically retries failed deliveries with exponential backoff
for up to 24 hours. If the Logic App or Storage Queue is temporarily
unavailable, events are not lost — they are retried until delivered.

7.2 Storage Queue Visibility Timeout
--------------------------------------
When the queue poller dequeues a message, it sets a 30-second visibility
timeout. If the Express API crashes before deleting the message, it becomes
visible again after 30 seconds and is reprocessed. This ensures at-least-once
delivery.

7.3 Azure Function Retry
--------------------------
The Function App is configured with the default retry policy. Failed
invocations are retried up to 3 times with exponential backoff before
being marked as failed in Application Insights.

7.4 Cosmos DB Optimistic Concurrency
--------------------------------------
Cosmos DB uses ETags for optimistic concurrency. When deactivating old
baselines before saving a new one, the replace operation uses the document's
ETag to prevent race conditions in concurrent baseline promotions.

7.5 Frontend Graceful Degradation
-----------------------------------
If the Express API is unreachable, the frontend falls back to demo mode
(useAzureScope hook catches the error and loads static demo data). The
Socket.IO connection silently retries up to 5 times before giving up.
The UI remains fully functional in demo mode for presentation purposes.

8. SECURITY CONSIDERATIONS
================================================================================

8.1 Credential Management
--------------------------
- No secrets are hardcoded in source code
- All secrets stored in .env files (excluded from git via .gitignore)
- Azure Function uses Managed Identity (no stored credentials)
- Express API uses DefaultAzureCredential (CLI credential locally,
  Managed Identity in production)

8.2 Azure Function Key
------------------------
The detectDrift function requires a function-level API key in the URL.
The Logic App and Event Grid subscription include this key in the endpoint URL.
The key is stored as an Azure Function App setting (not in source code).

8.3 Cosmos DB Access
---------------------
Cosmos DB is accessed via primary key (stored in .env). In production,
this should be replaced with Managed Identity + Cosmos DB RBAC roles
(Cosmos DB Built-in Data Contributor).

8.4 CORS Policy
----------------
The Express API currently allows all origins (cors({ origin: '*' })).
In production, this should be restricted to the frontend's domain.

8.5 Network Security
---------------------
All communication between services uses HTTPS/TLS. The Storage Account
enforces TLS 1.2 minimum. Cosmos DB enforces TLS 1.2 minimum.
The Function App and Express API should be placed behind Azure API Management
or an Application Gateway in a production deployment.

8.6 ARM Remediation Authorization
-----------------------------------
The remediate endpoint calls ARM to PUT resource configurations. This requires
Contributor role on the target resource group. In production, this should be
scoped to the minimum required permissions using a custom RBAC role.

 ## How Auto-Remediation Works

When you click "Remediate to Baseline", this is the exact sequence:

Frontend click
  → POST /api/remediate  { subscriptionId, resourceGroupId, resourceId }
    → Express API (remediate.js)
      1. Fetches golden baseline from Cosmos DB (baselines container, active=true)
      2. Fetches current live config from ARM API
      3. Computes diff (what changed)
      4. Calls ARM PUT with the baseline config
         → Azure Resource Manager applies the change to the actual resource
      5. Returns { remediated: true, changeCount, appliedBaseline, previousLiveState }
    → Frontend shows success panel with field-level diff


The ARM PUT call (beginCreateOrUpdateAndWait) is a real write operation to Azure — it literally changes the resource's configuration
in your Azure account to match the stored baseline. For example if TLS was changed from 1.2 to 1.0, it PUTs minimumTlsVersion: TLS1_2
back.
