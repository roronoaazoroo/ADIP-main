================================================================================
SECTION 1: PROJECT OVERVIEW
================================================================================

ADIP is a real-time Azure infrastructure drift detection, comparison, and
auto-remediation platform. It continuously monitors Azure resources, detects
any configuration change (drift) from a known-good "golden baseline", classifies
the severity of that drift, notifies administrators, and provides tools to
either approve remediation or roll back to any previously saved configuration.

Core Problem Solved:
  When teams manage Azure infrastructure, configurations drift over time —
  someone changes a firewall rule, modifies a tag, adjusts a storage quota.
  Without a system to track these changes, environments become inconsistent,
  security posture degrades, and compliance is hard to prove. ADIP solves this
  by treating every Azure resource configuration as versioned state.

Key Capabilities:
  1. Real-time change detection via Azure Event Grid
  2. Field-level diff showing exactly what changed, who changed it, and when
  3. Severity classification (Critical / High / Medium / Low)
  4. Email alerts with one-click Approve/Reject remediation links
  5. Auto-remediation for low-severity drift (no approval needed)
  6. Golden baseline management — promote any config as the reference state
  7. Configuration Genome — versioned snapshot history with rollback
  8. AI-powered drift explanation and severity re-classification (Azure OpenAI)
  9. Azure Policy compliance display alongside drift data
  10. User-based filtering of live activity feed
  11. CSV export of all activity

SECTION 2: SYSTEM ARCHITECTURE
================================================================================

The system has three layers:

  LAYER 1 — FRONTEND (React/Vite, port 5173)
    Single-page application. Communicates with the Express API via REST and
    Socket.IO for real-time updates.

  LAYER 2 — EXPRESS API (Node.js, port 3001)
    The central backend. Handles all Azure SDK calls, diff computation,
    baseline management, genome snapshots, email alerts, and Socket.IO rooms.
    Runs locally or can be deployed to Azure App Service.

  LAYER 3 — AZURE SERVICES
    Event Grid, Logic Apps, Function App, Storage Account, Communication
    Services, OpenAI — all managed Azure PaaS services.

Complete Data Flow:

  [User changes Azure resource in Portal/CLI]
         |
         v
  [Azure Resource Manager fires change event]
         |
         v
  [Event Grid Topic (adip-eg-topic)]
    Subscribed to: ResourceWriteSuccess, ResourceDeleteSuccess, ResourceActionSuccess
         |
         v
  [Logic App: adip-logic-app]
    Filters out: Failed status, read ops, list ops, /deployments/ paths
    Calls: Azure Function detectDrift via HTTP POST
    Also: Puts message in Storage Queue (resource-changes)
         |
         +---> [Azure Function: detectDrift]
         |       Fetches live ARM config
         |       Reads baseline from Blob Storage
         |       Runs field-level diff
         |       Writes drift record to Blob Storage
         |       Sends alert email directly via ACS
         |       POSTs to Express /internal/drift-event (if configured)
         |
         +---> [Storage Queue: resource-changes]
                 |
                 v
               [Express API: queuePoller]
                 Polls every 5 seconds
                 Fetches live ARM config (with child resources)
                 Diffs against liveStateCache or baseline
                 Auto-saves snapshot to baseline-genome
                 Emits resourceChange event via Socket.IO
                 |
                 v
               [Frontend: useDriftSocket hook]
                 Receives event
                 Displays in Live Activity feed
                 Shows: user, action, resource, field-level changes

SECTION 3: AZURE SERVICES — WHY EACH ONE WAS CHOSEN
================================================================================

--- 3.1 Azure Event Grid ---

What it does in ADIP:
  Receives ARM resource change notifications at the subscription level.
  Configured with event types: ResourceWriteSuccess, ResourceDeleteSuccess,
  ResourceActionSuccess. Delivers events to the Logic App via HTTP webhook.

Why Event Grid and not alternatives:
  - Azure Monitor Alerts: Only alerts on metrics (CPU, memory), not config changes.
  - Activity Log + Diagnostic Settings: Batch delivery, not real-time.
  - Azure Policy: Evaluates compliance but does not push change events.
  - Event Grid is the ONLY Azure service that fires an HTTP webhook within
    seconds of any ARM write operation. It is purpose-built for this use case.

Key configuration:
  Topic: adip-eg-topic (custom topic, West US 2)
  Subscription: adip-sub-logic routes to Logic App
  Subscription: adip-sub-changes routes to Storage Queue
  Schema: EventGridSchema (includes resourceUri, operationName, claims, caller)

--- 3.2 Azure Logic Apps (adip-logic-app) ---

What it does in ADIP:
  Acts as the event router between Event Grid and the Function App.
  Performs cheap pre-filtering before invoking the expensive Function App:
    - Drops events with status = Failed
    - Drops pure read operations (operationName contains 'read')
    - Drops /deployments/ resource paths (ARM internal noise)
  Then calls detectDrift Function App via HTTP POST with resourceId and
  subscriptionId extracted from the event.

Why Logic Apps and not alternatives:
  - Direct Event Grid to Function App: Possible, but Logic Apps provides
    visual filtering, retry logic, and run history for debugging without code.
  - Azure Service Bus: Adds cost and complexity for simple routing.
  - Logic Apps is serverless, has built-in HTTP action, and the filtering
    logic is maintainable without redeployment.

--- 3.3 Azure Logic Apps (adip-drift-alert) ---

What it does in ADIP:
  Secondary Logic App triggered by the Express API for drift alerts.
  Receives a drift record via HTTP POST, checks if severity is critical or high,
  and forwards to the Express API /alert/email endpoint.
  This provides an alternative alert path independent of the Function App.

Why a separate Logic App for alerts:
  Separation of concerns — the routing Logic App handles event ingestion,
  the alert Logic App handles notification. Each can be modified independently.

--- 3.4 Azure Function App (adip-func-001, Node 20) ---

What it does in ADIP:
  Runs the detectDrift function — the core drift detection engine.
  Triggered via HTTP POST from the Logic App.
  Steps:
    1. Receives resourceId and subscriptionId
    2. Fetches live ARM config using ResourceManagementClient
    3. Reads golden baseline from Blob Storage (baselines container)
    4. Runs the custom recursive diff engine (computeDiff)
    5. Classifies severity based on diff results
    6. If changes found: writes drift record to Blob Storage (drift-records)
    7. Sends alert email directly via Azure Communication Services
    8. POSTs to Express /internal/drift-event for Socket.IO broadcast

Why Azure Functions and not alternatives:
  - Always-on App Service: Overkill for event-driven processing.
  - Container Apps: More complex, not needed for a single function.
  - Azure Functions Consumption plan is serverless — zero cost when idle,
    scales automatically, and the HTTP trigger integrates directly with
    Logic Apps. Perfect for event-driven drift detection.

Deployment: zip deploy via 'func azure functionapp publish adip-func-001'
Runtime: Node.js 20 on Linux Dynamic (Consumption) plan

--- 3.5 Azure Storage Account (adipstore001) ---

What it does in ADIP — FOUR separate uses:

  a) Storage Queue (resource-changes):
     Receives ARM change events from Event Grid subscription adip-sub-changes.
     The Express API queuePoller reads from this queue every 5 seconds.
     Why Queue Storage: Simple, cheap, ordered message delivery. No need for
     Service Bus topics/subscriptions for a single consumer.

  b) Blob Container (baselines):
     Stores the golden baseline for each resource.
     Key format: base64url(resourceId).json
     Each blob contains: subscriptionId, resourceId, resourceState, promotedAt
     Why Blob Storage: O(1) point reads by key. No query needed — we always
     know the exact key from the resourceId. Cheaper than Cosmos DB for this.

  c) Blob Container (drift-records):
     Stores every detected drift event.
     Key format: ISO-timestamp_base64url(resourceId).json
     Timestamp prefix enables chronological listing without a database index.
     Why Blob Storage: Append-only write pattern, no updates needed.

  d) Blob Container (baseline-genome):
     Stores versioned configuration snapshots for the Genome feature.
     Key format: ISO-timestamp_base64url(resourceId).json
     Auto-populated on every detected change. Also manually saveable.

  e) Blob Container (drift-history):
     Reserved for future historical analytics.

Why Azure Blob/Queue Storage and not Cosmos DB:
  The original design used Cosmos DB but it was replaced with Blob Storage.
  Reason: Cosmos DB costs ~$25/month minimum (400 RU/s). Blob Storage costs
  fractions of a cent per GB. For this use case (key-value lookups by resourceId,
  append-only drift records), Blob Storage is functionally equivalent and
  dramatically cheaper. The only trade-off is no SQL-style queries — mitigated
  by using timestamp-prefixed blob names for chronological listing.

--- 3.6 Azure Communication Services (adip-comms + adip-email) ---

What it does in ADIP:
  Sends HTML email alerts for critical and high severity drift events.
  Email contains: resource name, severity, field-level diff table,
  Approve/Reject buttons linking back to the Express API.

Why Azure Communication Services and not alternatives:
  - SendGrid: Third-party, requires separate account and billing.
  - Office 365 SMTP: Requires Exchange license, complex auth.
  - Azure Communication Services is a first-party Azure service, integrates
    with DefaultAzureCredential, and provides a managed email domain
    (azurecomm.net) without needing a custom domain or DNS setup.
    The @azure/communication-email SDK makes it a 10-line integration.

Components:
  adip-comms: The Communication Services account (global)
  adip-email: The Email Services resource with AzureManagedDomain
  Sender: DoNotReply@d8515e4e-69a5-4147-a28a-8f41ebdabf18.azurecomm.net

--- 3.7 Azure OpenAI (adip-openai, East US) ---

What it does in ADIP — THREE AI features:

  a) Drift Explanation (explainDrift):
     Takes the list of changed fields and generates a 3-4 sentence plain-English
     explanation focused on security implications.
     Shown in the Comparison page as "AI Security Analysis".

  b) Severity Re-classification (reclassifySeverity):
     The rule-based classifier uses fixed paths (networkAcls, sku, etc.).
     The AI re-classifier considers context — e.g., changing a tag on a
     production Key Vault might be more significant than the rules suggest.
     Returns JSON: {severity: 'high', reasoning: 'one sentence'}
     If AI severity is higher than rule-based, AI wins.

  c) Remediation Recommendation (getRemediationRecommendation):
     Explains what reverting to baseline will do and whether it is safe.
     Shown before the user clicks Remediate.

  d) Anomaly Detection (detectAnomalies):
     Analyzes the last 50 drift records to find patterns:
     unusual actors, repeated changes to the same resource, off-hours activity.
     Returns up to 3 anomaly objects shown in the Dashboard sidebar.

Why Azure OpenAI and not OpenAI API directly:
  - Data residency: Azure OpenAI keeps data within the Azure tenant.
  - Auth: Uses the same Azure credential chain as all other services.
  - Compliance: Required for enterprise/government workloads.
  - The adip-gpt deployment uses GPT-4o (or equivalent) with temperature 0.3
    for consistent, deterministic outputs.

--- 3.8 Azure Policy (read-only, no deployment) ---

What it does in ADIP:
  The policyService.js uses PolicyInsightsClient to query compliance state
  for the selected resource or resource group.
  Results shown as a Policy badge in the Dashboard stats panel.
  Non-compliant count shown in the Comparison page navbar.

Why read-only:
  Creating policy assignments requires Owner or Policy Contributor role.
  ADIP only has Contributor access, so it reads existing policy compliance
  without creating or modifying assignments.

SECTION 4: BACKEND CODE WALKTHROUGH

--- 4.1 Entry Point: adip-backend/express-api/src/app.js ---

This file bootstraps the entire backend. Key things it does:

  require('dotenv').config({ path: '../../.env' })
    Loads all environment variables from the root .env file.
    The path resolves two levels up from src/ to reach the project root.
    This single .env file serves both the Express API and the frontend
    (Vite reads VITE_* prefixed variables automatically).

  const io = new Server(server, { cors: { origin: '*' } })
  global.io = io
    Creates a Socket.IO server attached to the same HTTP server as Express.
    Stored on global.io so any route or service can emit events without
    importing the io instance directly (avoids circular dependencies).

  io.on('connection', socket => {
    socket.on('subscribe', ({ subscriptionId, resourceGroup }) => {
      const room = resourceGroup ? subscriptionId:resourceGroup : subscriptionId
      socket.join(room)
    })
  })
    When the frontend connects, it sends a 'subscribe' event with its scope.
    The backend puts the socket into a room named by subscriptionId:resourceGroup.
    This means events are only sent to clients watching the relevant scope.
    A client watching all of sub-123 gets all events. A client watching
    sub-123:rg-adip only gets events for that resource group.

  startQueuePoller()
    Called once on server start. Begins polling the Storage Queue every 5s.

  POST /api/cache-state
    Called by the frontend immediately after Submit loads the config.
    Seeds the liveStateCache in queuePoller with the current resource state.
    This ensures the FIRST change event after Submit shows a proper diff
    (without this, the first event has no 'previous' state to diff against).

  POST /internal/drift-event
    Internal endpoint called by the deployed Function App.
    Receives a drift record, emits it to the correct Socket.IO room,
    and triggers sendDriftAlert for email notification.
    This is the bridge between the serverless Function App and the
    stateful Socket.IO server.

--- 4.2 Queue Poller: services/queuePoller.js ---

This is the real-time change detection engine running inside Express.

  parseMessage(msg)
    Decodes a base64 Storage Queue message back to JSON.
    Extracts: eventId, eventType, eventTime, resourceId, subscriptionId,
    resourceGroup, operationName, status, caller.

    Caller extraction:
      const claims = event.data?.claims || {}
      const caller = claims.name
                  || claims['http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name']
                  || event.data?.caller
                  || 'Unknown user'
    ARM Event Grid includes the Azure AD identity of whoever made the change
    in the claims object. The 'name' claim is the display name (e.g. 'Saksham Midha').
    The long URI claim is the fallback for older token formats.

    Child resource normalization:
      if (uriParts.length > 9) resourceUri = uriParts.slice(0, 9).join('/')
    When blob service settings change, Event Grid fires with a URI like:
    .../storageAccounts/foo/blobServices/default
    This has more than 9 path segments. We truncate to the parent resource
    (.../storageAccounts/foo) so we fetch the full parent config including
    all child resources.

  normalize(obj)
    Flattens _childConfig into the top-level object before diffing.
    Example: { properties: {...}, _childConfig: { blobServices: {...} } }
    becomes: { properties: {...}, blobServices: {...} }
    This ensures baselines saved before child config was added still diff
    correctly against current state.

  computeDiff(prev, curr, path, results)
    Recursive diff engine. Handles:
    - Null/undefined transitions (added/removed)
    - Nested objects (recurses into all keys)
    - Arrays (stable sort comparison to avoid false positives from key ordering)
    - Primitives (direct comparison)
    Produces results with: path, type, oldValue, newValue, sentence
    The 'sentence' field is a human-readable description:
      'changed "defaultAction" from "Allow" to "Deny"'
      'added tag 'environment' = "production"'
      'removed "days" (was 7)'

  enrichWithDiff(event)
    The main processing function called for each queue message:
    1. Fetches live ARM config (including child resources)
    2. Checks liveStateCache for previous state
    3. If no cache: falls back to stored baseline from Blob Storage
    4. Runs formatDiff(previous, current)
    5. Updates liveStateCache with current state
    6. If changes found: auto-saves snapshot to baseline-genome
    7. Returns enriched event with liveState, changes, changeCount, hasPrevious

  VOLATILE fields stripped before diffing:
    ['etag','changedTime','createdTime','provisioningState','lastModifiedAt',
     'systemData','_ts','_etag','primaryEndpoints','secondaryEndpoints',
     'primaryLocation','secondaryLocation','statusOfPrimary','statusOfSecondary',
     'creationTime']
    These are ARM-managed read-only fields that change without user action.
    Including them would flood the feed with meaningless noise.

--- 4.3 Azure Resource Service: services/azureResourceService.js ---

  CHILD_RESOURCES map:
    storageaccounts: [blobServices/default, fileServices/default,
                      queueServices/default, tableServices/default]
    sites: [config/web]
    These child resources hold config that ARM does NOT return on the parent GET.
    For storage accounts: soft delete settings, versioning, CORS, quota are
    all on blobServices/default, not on the storageAccounts resource itself.

  getResourceConfig(subscriptionId, resourceGroupName, resourceId)
    1. Fetches the parent resource via ARM GET
    2. Looks up CHILD_RESOURCES for the resource type
    3. Fetches all child resources in parallel (Promise.allSettled)
    4. Merges results into resource._childConfig
    5. Returns the enriched resource object
    Using Promise.allSettled (not Promise.all) means one failing child
    resource does not abort the entire fetch.

  getApiVersion(subscriptionId, provider, type)
    First checks the static API_VERSION_MAP (covers 35+ resource types).
    If not found, queries the ARM provider metadata to find the latest
    stable API version dynamically. Caches the result.
    This makes the system work with ANY Azure resource type, not just
    the ones explicitly listed.

--- 4.4 Blob Service: services/blobService.js ---

  blobKey(resourceId)
    Buffer.from(resourceId).toString('base64url') + '.json'
    Converts the ARM resource ID to a safe blob name.
    base64url encoding removes characters (/, +, =) that are problematic
    in blob paths. Deterministic: same resourceId always produces same key.

  driftKey(resourceId, ts)
    ISO-timestamp_base64url(resourceId).json
    Timestamp prefix means blobs sort chronologically when listed.
    This gives us time-ordered drift history without a database index.

  Genome functions:
    saveGenomeSnapshot(subscriptionId, resourceId, resourceState, label)
      Saves a timestamped snapshot to baseline-genome container.
      Called automatically by queuePoller on every detected change.
      Also called manually from the Genome page.

    listGenomeSnapshots(subscriptionId, resourceId, limit)
      Lists all snapshots for a resource, sorted newest-first.
      Scans the container (O(n) but acceptable for typical snapshot counts).

    getGenomeSnapshot(blobKey)
      Point read by blob key. O(1).

--- 4.5 Alert Service: services/alertService.js ---

  sendDriftAlert(record)
    Only sends for severity 'critical' or 'high'.
    Recipients from ALERT_RECIPIENT_EMAIL env var (comma-separated).

    buildDiffTable(changes)
      Generates an HTML table of all changed fields with color-coded
      type badges (MODIFIED=amber, ADDED=green, REMOVED=red).
      Old values shown with strikethrough, new values in green.

    buildApprovalButtons(record, baseUrl)
      Creates Approve and Reject buttons linking to:
      GET /api/remediate-decision?action=approve&token=<base64url>
      The token encodes: resourceId, resourceGroup, subscriptionId, detectedAt
      as base64url JSON. No signature — security relies on the token being
      delivered only via email to authorized recipients.

--- 4.6 Severity Classification ---

  classifySeverity(diffs) — used in queuePoller, detectDrift, compare.js

    CRITICAL if:
      - Any field is REMOVED (type === 'removed')
      - 3 or more tag changes in a single event

    HIGH if any changed path starts with:
      properties.networkAcls    (firewall rules, default action)
      properties.accessPolicies (Key Vault access)
      properties.securityRules  (NSG rules)
      sku                       (pricing tier change)
      location                  (region change)
      identity                  (managed identity)
      properties.encryption     (encryption key source)

    MEDIUM if: more than 5 fields changed (bulk change)

    LOW if: 1-5 non-critical field changes

  AI re-classification:
    After rule-based classification, reclassifySeverity() sends the changes
    to Azure OpenAI. If the AI returns a higher severity, it overrides the
    rule-based result. The AI considers context the rules cannot:
    e.g., changing a tag on a production resource during off-hours.

--- 4.7 Remediation Flow ---

  LOW severity (auto-remediate):
    Frontend calls POST /api/remediate
    Express fetches baseline, fetches live state, diffs them
    Calls armClient.resources.beginCreateOrUpdateAndWait() with baseline state
    ARM PUT reverts the resource. No email sent.

  CRITICAL/HIGH/MEDIUM severity (approval flow):
    Frontend calls POST /api/remediate-request
    Express calls sendDriftAlert() which sends HTML email with Approve/Reject
    Admin clicks Approve link in email
    Browser hits GET /api/remediate-decision?action=approve&token=...
    Express decodes token, fetches baseline, applies ARM PUT
    Returns HTML confirmation page

  Reject flow:
    Admin clicks Reject link
    Express fetches current live state, saves it as the new baseline
    The drift is "accepted" — current state becomes the new golden baseline

--- 5.6 Socket.IO Hook: src/hooks/useDriftSocket.js ---

  Connects to VITE_SOCKET_URL (defaults to VITE_API_BASE_URL without /api)
  On connect: emits 'subscribe' with subscriptionId and resourceGroup
  Listens for 'resourceChange' events

  Event gating:
    if (!isSubmittedRef.current) return
    Events are dropped until the user has pressed Submit.
    This prevents stale events from appearing before the user has context.

  Scope filtering:
    matchesSub: event.subscriptionId === scope.subscriptionId
    matchesRG:  !scope.resourceGroup || event.resourceGroup === scope.resourceGroup
    matchesRes: !scope.resourceId || event.resourceId matches scope.resourceId
    Only events matching the current scope are shown.

  Deduplication:
    Each event gets a _clientId: eventId-random to prevent React key collisions
    when the same event arrives twice (e.g., from both queue paths).

--- 5.7 Azure Scope Hook: src/hooks/useAzureScope.js ---

  Resource group IDs use the name (not ARM ID) for simplicity:
    id: rg.name ?? rg.id
  This is why resourceGroupId throughout the app is 'rg-adip' not the full ARM path.

--- 5.8 JsonTree Component: src/components/JsonTree.jsx ---

SECTION 6: CONFIGURATION GENOME — DETAILED EXPLANATION
================================================================================

The Genome system treats every Azure resource configuration as versioned state,
similar to how Git treats source code. Every configuration state is a commit.

Storage:
  Container: baseline-genome in adipstore001
  Key format: 2026-04-07T10-30-00-000Z_<base64url(resourceId)>.json
  Each blob contains:
    subscriptionId, resourceId, resourceState (full ARM config),
    label (human-readable description), savedAt (ISO timestamp)

Auto-save trigger:
  In queuePoller.js, after enrichWithDiff() detects changes:
    if (changes.length > 0) {
      saveGenomeSnapshot(subscriptionId, resourceId, current,
        'auto: N change(s) by <caller>')
    }
  This means every real change automatically creates a snapshot.
  The label records who made the change and how many fields changed.

Manual save:
  User clicks '+ Save Snapshot' on the Genome page with an optional label.
  Calls POST /api/genome/save which fetches current live config and saves it.
  Useful for: 'before maintenance window', 'post-deployment baseline', etc.

Set as Baseline (promote):
  POST /api/genome/promote { subscriptionId, resourceGroupId, resourceId, blobKey }
  Reads the snapshot from baseline-genome/
  Writes it to baselines/ (overwrites the golden baseline)
  The Comparison page will now diff against this snapshot.

Rollback:
  POST /api/genome/rollback { subscriptionId, resourceGroupId, resourceId, blobKey }
  Reads the snapshot
  Strips VOLATILE fields (etag, provisioningState, etc.)
  Calls armClient.resources.beginCreateOrUpdateAndWait() with snapshot state
  This is a synchronous ARM PUT — waits for Azure to confirm the change.
  The resource is now in the exact state it was when the snapshot was taken.

Difference from Remediation:
  Remediation reverts to the GOLDEN BASELINE (the single approved reference).
  Rollback reverts to ANY snapshot in the genome history.
  Rollback is more flexible — you can go back to any point in time.
  Remediation is the compliance action — restoring the approved state.

  baselines/
    One blob per resource. Key: base64url(resourceId).json
    Content: { subscriptionId, resourceGroupId, resourceId,

               liveState, baselineState, differences, severity,
               changeCount, detectedAt, aiExplanation, aiSeverity }
    Written by: detectDrift Function App, POST /compare

  baseline-genome/
    One blob per snapshot. Key: timestamp_base64url(resourceId).json
    Content: { subscriptionId, resourceId, resourceState, label, savedAt }
    Written by: queuePoller (auto, on every change),
                POST /genome/save (manual)

  drift-history/
    Reserved for future use. Currently empty.

  function-releases/
    Managed by Azure Functions deployment. Contains zip packages.
    Do not modify manually.

SECTION 10: KNOWN LIMITATIONS AND FUTURE IMPROVEMENTS
================================================================================

  1. Authentication
     Current: Local username/password with hardcoded DUMMY_USERS.
     Limitation: Not suitable for production. Credentials in source code.
     Fix: Implement MSAL SSO using VITE_AZURE_CLIENT_ID. Requires Azure AD
     App Registration (Owner access). The auth.js service is already prepared.

  2. Express API is local-only
     Current: Express runs on localhost:3001.
     Limitation: Email approval links point to localhost. Function App cannot
     reach Express for Socket.IO broadcast.
     Fix: Deploy Express to Azure App Service. Update EXPRESS_PUBLIC_URL and
     EXPRESS_API_URL in both .env and Function App settings.

  3. Blob Storage listing is O(n)
     Current: getDriftRecords and listGenomeSnapshots scan all blobs.
     Limitation: Slow with thousands of records.
     Fix: Add Azure Cognitive Search index on blob metadata, or migrate
     drift-records to Azure Table Storage for indexed queries.

  4. No authentication on API endpoints
     Current: All /api/* endpoints are open (CORS allows all origins).
     Limitation: Anyone who can reach port 3001 can call any endpoint.
     Fix: Add JWT middleware. When MSAL SSO is implemented, validate the
     Azure AD access token on every request.

  5. liveStateCache is in-memory
     Current: Cache is lost on Express restart.
     Limitation: First event after restart shows no diff (falls back to baseline).
     Fix: Persist liveStateCache to Redis Cache or Azure Table Storage.

  6. Function App Node.js version
     Current: Node 20 (reaches end-of-life April 30, 2026).
     Fix: Upgrade to Node 24 in Function App settings.

  7. Genome rollback has no diff preview
     Current: Rollback applies immediately after confirm dialog.
     Improvement: Show a diff between current state and snapshot before
     applying, similar to the Comparison page.

  8. No multi-subscription genome
     Current: Genome snapshots are per-resource within one subscription.
     Improvement: Cross-subscription genome for enterprise environments.

SECTION 12: SECURITY CONSIDERATIONS
================================================================================

  1. Secrets in .env
     The root .env contains live Azure keys (Storage, Event Grid, OpenAI, ACS).
     .gitignore excludes .env files. Never commit .env to version control.
     Rotate keys if accidentally exposed (e.g., in git history).

  2. Function App key in Logic App
     The Logic App workflow.json stores the Function App key.
     This file IS committed to git (with the key replaced by YOUR_FUNCTION_KEY).
     The actual key is only in the deployed Logic App definition.
     Rotate the Function App key periodically via:
       az functionapp keys set --name adip-func-001 --resource-group rg-adip
         --key-type functionKeys --key-name default --key-value ""

  3. Email approval token
     The remediation approval token is base64url-encoded JSON (not signed).
     Security relies on: token delivered only to authorized email recipients,
     token contains detectedAt timestamp (can add expiry check),
     HTTPS in production prevents interception.
     For production: sign the token with HMAC-SHA256 using a secret key.

  4. Contributor access scope
     ADIP uses DefaultAzureCredential which picks up the az login session.
     Contributor access allows: read all resources, write/modify resources,
     deploy Function Apps, manage Logic Apps.
     Contributor does NOT allow: creating role assignments, modifying policies,
     accessing Key Vault secrets (requires separate Key Vault access policy).

  5. CORS
     Express API has CORS set to origin: '*'.
     For production: restrict to the frontend domain only.
